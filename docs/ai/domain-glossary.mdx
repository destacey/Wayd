# Domain Glossary

This glossary defines the key terms and concepts used throughout Wayd. It is designed to help AI agents and developers understand the domain language.

## Organization

| Term | Definition |
|------|-----------|
| **Team** | A group of people that work together to deliver value. Has a Name, Code (unique 2-10 char identifier), Active Date, and Operating Model. Can be a member of a Team of Teams. |
| **Team of Teams** | A higher-level organizational unit that contains child teams and other teams of teams, forming the organizational hierarchy. Provides rollup visibility. Does not have backlogs or sprints. |
| **Team Code** | A unique 2-10 character uppercase alphanumeric identifier for a team (e.g., `PLATFORM`, `MOBILE`). |
| **Operating Model** | Defines how a team works: Methodology (Scrum or Kanban) and Sizing Method (Story Points or Count). Has a date range creating a history of changes. |
| **Methodology** | The team's delivery approach. Scrum = sprint-based with time-boxed iterations. Kanban = continuous flow from product backlog. |
| **Sizing Method** | How the team estimates work. Story Points = relative sizing. Count = each item counts as 1. |
| **Team Membership** | A date-ranged parent-child relationship between a team and a Team of Teams. States: Past, Active, Future. Only one active parent per team. |
| **Employee** | An individual in the organization. Can be a member of multiple teams. May be synced from Azure AD via Microsoft Graph. |
| **Functional Organization Chart** | A visualization of the complete team hierarchy as of a specific date. |

## Planning

| Term | Definition |
|------|-----------|
| **Planning Interval (PI)** | A fixed timeframe (typically 8-12 weeks) during which teams deliver value toward committed objectives. Contains iterations, objectives, and team associations. States: Future, Active, Completed. |
| **Iteration** | A time-boxed period within a PI. Has a Name, Category (Development or Innovation and Planning), and Date Range. Aggregates sprint metrics across all mapped team sprints. |
| **Sprint** | A short time-boxed period (typically 2 weeks) owned by a team. Type: Sprint or Iteration. State: Future, Active, Completed. Can be Owned (Wayd) or Managed (external system). |
| **Sprint Mapping** | The association of a team's sprint to a PI iteration. One sprint per team per iteration. Enables cross-team aggregate views. |
| **Planning Interval Objective** | A team's commitment to deliver a specific outcome within a PI. Has Status (NotStarted, InProgress, Completed, Canceled, Missed), IsStretch flag, and optional Health Check. |
| **Stretch Objective** | An aspirational goal that is not counted toward predictability. Contrasts with committed objectives. |
| **Predictability** | The percentage of committed (non-stretch) objectives completed in a PI. Formula: (Completed committed / Total committed) × 100. Stretch objectives excluded. |
| **Objectives Locked** | A PI flag that prevents creating new objectives or changing the stretch flag. Used after PI planning is complete. |
| **Plan Review** | A team-by-team view of the PI for planning ceremonies. Shows objectives and risks per team with list and timeline views. |
| **Risk** | A potential circumstance that could impact delivery. Tracked with Status (Open/Closed), ROAM Category, Impact, Likelihood, Exposure, Assignee, and Follow-Up Date. |
| **ROAM** | Risk categorization model: Resolved (eliminated), Owned (being managed), Accepted (acknowledged), Mitigated (impact reduced). |
| **Exposure** | Calculated risk severity from Impact × Likelihood. Low (less than 4), Medium (equals 4), High (greater than 4). Each dimension is Low=1, Medium=2, High=3. |
| **Roadmap** | A visual timeline for planning work. Has Visibility (Public/Private), Managers (min 1), and hierarchical Items (Activities, Milestones, Timeboxes). Can be copied. Optionally has a **Color palette** — an ordered set of named colors (one optional default) used to color activities and drive the timeline legend. |
| **Activity** | A roadmap item with a date range that can contain child items (other activities, milestones, timeboxes). Ordered within parent. |
| **Milestone** | A roadmap item representing a single point in time (key date or achievement). |
| **Timebox** | A roadmap item representing a fixed time period (e.g., a sprint or phase). |
| **Planning Poker** | Collaborative estimation using estimation scales and real-time voting via SignalR. Behind `planning-poker` feature flag. |
| **Estimation Scale** | A named set of values for estimation (e.g., Fibonacci: 1, 2, 3, 5, 8, 13). Minimum 2 values. |
| **Poker Session** | A real-time estimation event with a facilitator, estimation scale, and rounds. Immediately Active on creation. |
| **Poker Round** | One estimation within a session. Workflow: Voting → Revealed → Accepted. Can be Reset to re-vote. |

## Planning Metrics

| Term | Definition |
|------|-----------|
| **Total** | Sum of all work items (or story points) in a sprint. |
| **Completed** | Items where status category is Done OR Removed. Includes both. |
| **Remaining** | Total minus Completed. |
| **Completion Rate** | (Completed / Total) × 100. Target: 80%. |
| **Velocity** | Completed count (story points or items delivered). |
| **Average Cycle Time** | Mean days from Activated to Done for eligible items. Only Done items (not Removed) with both timestamps. |
| **Missing Story Points** | Count of items with null or zero story points. Indicates estimation gaps. |
| **Time to Start** | Days from work item creation to activation (how long work waited). |
| **Cycle Time** | Days from activation to completion. Formula: DoneTimestamp − ActivatedTimestamp. |
| **Lead Time** | Days from creation to completion (total request-to-delivery time). |

## Work Management

| Term | Definition |
|------|-----------|
| **Work Item** | The fundamental unit of work. Has a Key (WorkspaceKey-Number), Title (immutable), Type, Status, StatusCategory, Priority, StoryPoints, Assignee, Team, Parent, Sprint, Project. |
| **Work Item Key** | Unique identifier in format `WorkspaceKey-Number` (e.g., `PLAT-1234`). Workspace key is 2-20 uppercase alphanumeric starting with a letter. |
| **Workspace** | A container for work items using a specific work process. Has a Key, Name, Ownership (Owned/Managed), and IsActive flag. |
| **Work Process** | A configuration linking work types to workflows. Assigned to workspaces. Requires at least one work type. Each type appears once. |
| **Work Process Scheme** | A mapping of a work type to a workflow within a work process. |
| **Work Type** | A category of work item (e.g., Epic, Feature, User Story, Task, Bug). Name is immutable. Assigned to a tier and level. |
| **Work Type Tier** | Grouping of work types by purpose: Portfolio (container, multi-level), Requirement (team work), Task (child items), Other (non-standard). |
| **Work Type Level** | Hierarchical ordering within a tier. System defaults: Epics (Portfolio L1), Features (Portfolio L2), Stories (Requirement L1), Tasks (Task L1). Custom Portfolio levels can be added. |
| **Workflow** | A sequence of statuses defining how work items progress. Contains a workflow scheme mapping statuses to categories and order. |
| **Workflow Scheme** | A mapping of a work status to a position in a workflow, with a status category and order. |
| **Work Status** | A specific state within a workflow (e.g., "New", "In Progress", "Done"). Name is immutable. Reusable across workflows. |
| **Work Status Category** | Fixed normalization set: Proposed (not started), Active (in progress), Done (completed), Removed (removed without completion). Enables cross-workflow metrics. |
| **Work Item Dependency** | A predecessor/successor relationship between work items. Tracks State (ToDo/InProgress/Done/Removed/Deleted), Health (Healthy/AtRisk/Unhealthy), and Scope (IntraTeam/CrossTeam). |
| **Dependency Health** | Automatically calculated from planned dates. Healthy = predecessor before successor. AtRisk = unplanned. Unhealthy = predecessor after successor or scheduling conflict. |
| **Work Item Revision** | An audit trail entry: revision number, who, when, and field-level changes (old/new values). |
| **Work Item Hierarchy** | Parent-child relationship. Only Portfolio tier types can be parents. Parent must be higher level than child. |
| **Backlog** | A team's unified view of all assigned work items across all workspaces. Visible on the team's Backlog tab. |

## Project Portfolio Management

| Term | Definition |
|------|-----------|
| **Portfolio** | Top-level container for programs, projects, and strategic initiatives. Status lifecycle: Proposed → Active → OnHold ↔ Active → Closed → Archived. May assign one active scoring model to enable project priority scoring. Owns the project ranking for all projects in the portfolio. |
| **Program** | Groups related projects within a portfolio. Status: Proposed → Active → Completed/Cancelled. Only active programs accept new projects. |
| **Project** | A timed endeavor with scope, timeline, and deliverables. Key is 2-20 uppercase alphanumeric. Status: Proposed → Approved (requires lifecycle) → Active (requires dates) → Completed/Cancelled. |
| **Project Key** | A unique 2-20 character uppercase alphanumeric code identifying a project (e.g., `APOLLO`). Can be changed (cascades to task keys). |
| **Project Lifecycle** | A template of phases that projects go through. States: Proposed (editable) → Active (locked) → Archived. Must have min 1 phase to activate. |
| **Project Lifecycle Phase** | A template phase within a lifecycle (e.g., Initiation, Planning, Execution, Closing). Ordered. Immutable after lifecycle activation. |
| **Project Phase** | A concrete instance of a lifecycle phase on a project. Has Status (NotStarted/InProgress/Completed/Cancelled), Date Range, Progress (0-100%), and Assignee roles. Name copied from template (not editable). |
| **Project Task** | An actionable work item within a project phase. Key: ProjectKey-Number. Type: Task (date range, 0-100% progress) or Milestone (single date, 0 or 100% only). Supports hierarchy (WBS) and Finish-to-Start dependencies. |
| **Project Rank** | Opaque fractional sort key used to order projects within a portfolio. Every project receives a rank on creation. Users see the derived 1-based display position, not the raw rank value. |
| **Portfolio Ranking Board** | Portfolio detail tab that shows Proposed, Approved, and Active projects in rank order. Portfolio Owners/Managers can drag projects to reorder them; score columns come from the current portfolio scoring model when assigned. |
| **Project Priority Score** | The latest score recorded for a project using its portfolio's assigned scoring model. Shows the primary score value, scorer, timestamp, and model name. |
| **Project Score Snapshot** | An immutable score history entry for a project. Stores the scoring model id/key/name, criterion ratings, computed outputs, scorer, timestamp, and per-project sequence so prior scores remain meaningful after model or portfolio assignment changes. |
| **Task Priority** | Low, Medium, High, or Critical. Shown as color indicators in the UI. |
| **Task Dependency** | Finish-to-Start relationship between project tasks. Soft-deleted (marked as removed, not permanently deleted). |
| **Strategic Initiative** | A high-level strategic effort within a portfolio, tracked with KPIs. Status: Proposed → Approved → Active → OnHold ↔ Active → Completed/Cancelled. |
| **KPI** | Key Performance Indicator on a strategic initiative. Has StartingValue, TargetValue, ActualValue (from latest measurement), Direction (Increase/Decrease), Checkpoints, and Measurements. |
| **KPI Checkpoint** | A planned target value at a specific date within a KPI. Has TargetValue and optional AtRiskValue threshold. |
| **KPI Measurement** | An actual recorded value at a specific date. Most recent becomes the KPI's ActualValue. |
| **Expenditure Category** | Classifies project spending. Has IsCapitalizable, RequiresDepreciation, AccountingCode. Settings frozen after activation. |
| **My Projects Dashboard** | Personalized view of projects where the user has a role. Two-panel layout: portfolio-grouped project cards (left) and selected project detail with phase timeline and task summary (right). |

## Strategic Management

| Term | Definition |
|------|-----------|
| **Vision** | The desired future state of the organization. Only ONE can be Active at any time. States: Proposed → Active (start date) → Archived (end date). Archived visions cannot overlap in dates. |
| **Strategy** | The approach for achieving the vision. States: Draft → Active → Completed → Archived. Independent of visions. |
| **Strategic Theme** | A high-level organizational priority guiding investment decisions. States: Proposed → Active → Archived. Only Proposed deletable. Cross-domain: tags Projects, Programs, Strategic Initiatives in PPM. |

## Identity & Access

| Term | Definition |
|------|-----------|
| **Identity Provider (OIDC Provider)** | A configured single sign-on source (Microsoft Entra ID or a generic OIDC issuer), stored as a row in `Identity.OidcProviders`. The database is the sole source of truth; providers are managed via Settings → Identity Providers. |
| **UserIdentity** | The link between an `ApplicationUser` and a login provider, keyed by `(Provider, ProviderTenantId, ProviderSubject)`. At most one active row per user; every authentication path resolves through this table. |
| **Role** | A named permission group assigned to users. One system role (`Admin`) is seeded and protected; admins can create custom roles. |
| **Auto-Registration (JIT Provisioning)** | Creating a Wayd account automatically the first time an unknown user signs in through an identity provider. Governed by a per-provider registration policy; **off by default** (secure by default). |
| **Registration Policy** | The per-provider settings that govern auto-registration: `AllowAutoRegistration` (master switch), `RequireEmployeeRecord` (employee-match gate), and `DefaultRoleId` (role for new users). Modeled as a `RegistrationPolicy` value object whose dependent fields are only meaningful while auto-registration is enabled. |
| **Employee-Record Gate** | The `RequireEmployeeRecord` policy: when on, auto-registration is restricted to users whose email matches an existing `Employee`. Turning it off grants accounts to anyone who can authenticate through the provider. |
| **Default Role** | The role assigned to auto-registered users for a provider (`DefaultRoleId`); required when auto-registration is enabled (no implicit fallback). Protected by a foreign key — a role pinned as a default cannot be deleted until the provider is repointed. |
| **First-User Bootstrap** | The first user to sign in to an empty deployment is always created and made `Admin`, bypassing the registration policy so a fresh install cannot lock itself out. |
| **Identity Migration** | Admin-staged moves that preserve `ApplicationUser.Id`: Entra tenant migration, cross-provider migration (OIDC→OIDC or Wayd→OIDC), and immediate OIDC→local conversion. |

## Cross-Cutting Concepts

| Term | Definition |
|------|-----------|
| **Ownership** | Whether an object is Owned (Wayd-managed), Managed (external system, read-only in Wayd), or System (immutable Wayd defaults). |
| **Link** | A cross-entity association between any Wayd object and an external URL. Simple name + URL pair. |
| **Connection** | A configured integration with an external system via a Connector (e.g., Azure DevOps). Includes auth and endpoint config. |
| **Connector** | A configurable integration type that enables a specific external service to plug into Wayd (e.g., AzureDevOps, Entra, AzureOpenAI). Each connector declares the capabilities it supports (WorkItems, People, AiProvider); a multi-surface connector can support several through one connection. Multiple connections per connector type are allowed. |
| **Connector Capability** | A discrete integration surface a connector supports, named for the resource or service provided. `WorkItems` routes through `WorkSyncRunner`; `People` routes through `PeopleSyncRunner`; `AiProvider` connectors aren't synced. The capability — not the specific connector — drives which background job and Sync History view applies. Each capability carries a display category (its `Display(GroupName)`, e.g. "Work Management") that groups the connector picker UI. |
| **Feature Flag** | A toggle controlling feature visibility. Defined in code, stored in database, managed via Settings UI. New flags seeded as disabled on startup. |
| **Health Check** | A status indicator (Healthy, AtRisk, Unhealthy/Critical) with expiration. Used on objectives, initiatives, and other entities. |
| **Scoring Model** | A reusable formula-driven model for rating items. Contains criteria, optional rating scales, ordered output formulas, and exactly one primary output. Lifecycle: Proposed (editable) → Active (assignable/locked) → Archived. |
| **Scoring Criterion** | A scoring-model input that a user rates. Has a unique token for formulas, optional description, optional weight, and optional rating scale. |
| **Scoring Output** | An ordered formula in a scoring model. Outputs can reference criterion tokens and earlier output tokens; one output is primary and becomes the score. |
| **Rating Scale** | A named set of ordered rating levels on a scoring model. Criteria can reference a scale so raters select standardized labels/values instead of entering a raw number. |
| **SystemContext** | Identifies which domain references a work item (e.g., Goals, Projects). Used for work item references. |

## Technical Terms

| Term | Definition |
|------|-----------|
| **Bounded Context** | Each domain service (Work, Organization, etc.) is a bounded context with its own ubiquitous language and models. |
| **Aggregate Root** | The primary entity ensuring consistency in a cluster of related objects. Feature folders organized by aggregate root. |
| **Value Object** | An immutable object defined by attributes, not identity (e.g., WorkItemKey, ProjectKey, FlexibleDateRange, Progress). |
| **Domain Event** | A notification that something significant happened in a domain, published for cross-context communication (e.g., TeamCreatedEvent, StrategicThemeUpdatedEvent). |
| **Command** | A request to change state (write operation). Handled via MediatR. Returns `Result&lt;T&gt;`. |
| **Query** | A request to read state (read operation). Handled via MediatR. Returns `Result&lt;T&gt;`. |
| **Result** | A functional return type (`Result&lt;T&gt;`) from CSharpFunctionalExtensions. Explicitly represents success or failure without exceptions. |
| **Work Process Scheme** | Links a work type to a workflow within a work process. |
| **Workflow Scheme** | Maps a work status to a position (order) and category within a workflow. |
| **FlexibleDateRange** | A value object with required start date and optional end date. Used by portfolios, programs, and other entities with open-ended timelines. |
| **LocalDateRange** | A value object with required start and end dates. Used by PIs, iterations, sprints, and roadmaps. |
