# Administration

The Administration area provides tools for configuring Wayd, managing users, controlling feature visibility, and connecting to external systems.

## User Management

### Users

User accounts control access to Wayd. Each user has:
- **Display Name** and **Email**
- **Login Provider** — How the user authenticates (set at creation, immutable):
  - **Microsoft Entra ID** — Single sign-on via Azure Active Directory
  - **Wayd (Local)** — Username/password authentication with JWT tokens
- **Roles** — Permission groups that control what the user can see and do
- **Status** — Active, Inactive, or Locked

**Account operations:**
- **Activate** — Enable a user account
- **Deactivate** — Disable a user account
- **Unlock** — Unlock an account locked due to failed login attempts
- **Reset Password** — Admin-initiated password reset (local auth only, forces change on next login)
- **Change Identity Provider** — Stage a move to a different OIDC provider, or from a local account to an OIDC provider (see below)
- **Convert to Local Account** — Immediately switch an OIDC user to password-based login (see below)

Local users created by an administrator have a **Must Change Password** flag that requires them to set a new password on first login.

#### Creating users

Administrators create accounts from **Settings → User Management → Users → Create User**. This is the path for provisioning a user before their first sign-in — for example when a provider's auto-registration is off, or for local (Wayd) accounts.

The form captures the user's details, their **Login Provider** (set here and immutable afterward), a password for local accounts, and their **Roles**.

**Things to know:**

- **At least one role is required.** You must assign one or more roles when creating a user — there is no default role applied automatically. A user with no roles has no permissions, so the form won't submit until at least one is selected.
- **Login provider is permanent.** The provider chosen at creation can't be changed later through editing; use **Change Identity Provider** to move a user between providers.
- Roles can be adjusted any time afterward from the user's detail page.

#### Tenant migration

When your organization moves users from one Microsoft Entra ID tenant to another, you stage the migration **in bulk from the provider page**. The rebind completes automatically the next time each user signs in from the new tenant — they don't need to do anything special, and their `UserId`, permissions, and links to other records (work items, audit history, etc.) are all preserved.

This lives on the provider's detail page (**Settings → Identity Providers → \[provider]**), and only appears for a Microsoft Entra ID provider configured with **two or more** allowed tenants (there must be tenants to move between).

**Workflow:**

1. On the provider detail page, open the **Actions** menu and pick **Migrate Users to New Tenant**.
2. In the dialog, choose a **source** tenant and a **target** tenant (both from the provider's allowed-tenant list). Wayd lists the users currently on the source tenant who don't already have a migration in progress.
3. Select the ones to move (up to 500 at a time) and click **Stage migration**. Wayd reports how many were staged and, if any selected users no longer qualify, how many were skipped.
4. The **Active Migrations** tab on the provider page lists everyone with a staged-but-incomplete migration, showing their source and target tenant and when it was staged.
5. Nothing else changes until each user signs in. Sign-ins from the *old* tenant continue to work normally.
6. When a user next signs in from the new tenant, Wayd detects the staged migration and atomically swaps their identity binding. The user notices nothing; afterward their **Identity History** shows the old binding (Inactive, reason: Tenant Migration) alongside the new active one.

**Things to know:**

- **Stage before the user signs in from the new tenant.** If you skip staging and the user signs in from the new tenant first, Wayd creates a brand-new account with a separate user ID — there's no in-app way to merge it back with the original; it requires manual database work to recover.
- Tenant migration is **Microsoft Entra ID only**. For moving a user between different OIDC providers, use **Change Identity Provider** instead.
- The selectable list excludes users who already have a migration pending. To retarget such a user, cancel their pending migration first (below).
- The **Active Migrations** tab is read-only. To cancel a single user's pending migration, open that user's detail page (**Settings → User Management → Users → \[user]**) and pick **Cancel Pending Migration** from the **Actions** menu. There is no bulk-cancel.

#### Change Identity Provider

Moves a user from one OIDC provider to another, or from a local (Wayd) account to an OIDC provider. Like tenant migration, the change is **deferred** — the user's current login continues to work until they sign in via the target provider for the first time, at which point the rebind happens automatically.

This action is available for any user (including local accounts) when at least one enabled OIDC provider is configured that differs from their current one.

**Workflow:**

1. On the user detail page, open the **Actions** menu and pick **Change Identity Provider** (or **Replace Pending Provider Migration** if one is already staged).
2. Select the target provider from the dropdown of enabled OIDC providers and confirm.
3. The user's current login continues to work as normal.
4. When the user next signs in via the target provider, Wayd matches them by email, atomically deactivates their old identity, inserts the new one, and updates their login provider. The user notices nothing.
5. The **Identity History** tab records the old binding as Inactive (reason: Provider Relinked).

**Things to know:**

- **Stage before the user signs in via the new provider (Entra targets only).** If the target provider is Microsoft Entra ID and the user signs in there before staging, Wayd creates a new account — there's no in-app merge path. For GenericOidc targets (Google, Okta, Auth0, Keycloak, etc.) this is not a risk: unknown users are rejected at the token-exchange boundary until a migration is staged for them.
- For users migrating **from a local account**: their password hash is cleared when the rebind completes, since the Wayd identity is deactivated. If the migration needs to be reversed, use **Convert to Local Account**.
- The action is hidden when only one OIDC provider is configured, or when the user is already on the only available provider.
- Re-staging with a different target provider silently replaces the previous one. Cancellation is available and idempotent.

#### Convert to Local Account

Converts an OIDC user to a local (password-based) account **immediately** — no deferred rebind. The admin sets a temporary password; the user is required to change it on their next login.

**Workflow:**

1. On the user detail page, open the **Actions** menu and pick **Convert to Local Account**.
2. Enter and confirm a temporary password, then confirm the action.
3. The user's OIDC identity is immediately deactivated and replaced with a local Wayd identity. Their next login must use the temporary password and they will be prompted to set a new one.

**Things to know:**

- This action is immediate and cannot be undone via the UI. To move the user back to OIDC, use **Change Identity Provider** to stage a new deferred migration.
- The user's `UserId`, permissions, and all other records are preserved — only the authentication method changes.
- The temporary password must meet the deployment's password policy (minimum 8 characters).

### Identity Providers

Single sign-on providers are configured in **Settings → Identity Providers** (available once the `IdentityProviders` feature flag is enabled). Beyond the connection details — issuer, client ID, audience, scopes, and (for Microsoft Entra ID) the allowed tenants — each provider carries a **registration policy** that controls whether and how new users are created when someone signs in for the first time.

**Registration policy fields:**

- **Allow automatic user registration** — The master switch. **Off by default.** When off, a person who authenticates successfully but has no Wayd account yet is turned away, and an administrator must create their account first. When on, a matching account is created automatically on first sign-in (subject to the options below).
- **Require matching employee record** — Shown only when auto-registration is on; defaults to on. While on, only people whose email matches an existing employee can self-register. Turning it off is a deliberate loosening — **anyone** who can authenticate through this provider will be granted an account — so the form shows a warning when you do.
- **Default role for new users** — Shown only when auto-registration is on, where it is **required**. The role assigned to accounts created through this provider — pick any role, system or custom. There is no implicit fallback; you must choose one explicitly.

**What happens on first sign-in:**

| Auto-registration | Employee record | Result |
|---|---|---|
| Off | — | Sign-in rejected; an admin must pre-create the account. |
| On | Required, no match | Sign-in rejected ("restricted to users with an employee record"). |
| On | Required, match found | Account created, assigned the default role. |
| On | Not required | Account created for anyone who authenticates, assigned the default role. |

**Things to know:**

- **The first user is special.** The very first person to sign in to a brand-new deployment always becomes an administrator, regardless of policy — this can't lock you out of an empty system.
- **Upgrades are secure by default.** Existing providers have auto-registration turned **off** after upgrading to this version; re-enable it deliberately per provider.
- **A role in use can't be deleted.** If a role is set as a provider's default, attempting to delete it is blocked with a clear message until you repoint the provider to a different role.

### Roles

Roles define permission groups. Users are assigned to roles to control access to features, pages, and actions throughout the application.

Permissions follow a hierarchical pattern:
- `Permissions.\{Domain\}.View` — View entities
- `Permissions.\{Domain\}.Create` — Create new entities
- `Permissions.\{Domain\}.Update` — Modify existing entities
- `Permissions.\{Domain\}.Delete` — Remove entities

### Personal Access Tokens

Users can generate **Personal Access Tokens (PATs)** for API access. These are managed from the user's profile page.

## Feature Management

Wayd uses **feature flags** to control the visibility of features across the application. Feature flags allow new functionality to be deployed but hidden until ready for use.

### How Feature Flags Work

1. Flags are **defined in code** by developers
2. On application startup, new flags are **automatically seeded** into the database — usually disabled, though some (such as replacement UIs being rolled out) are seeded enabled. Seeding only sets a flag's **initial** state; your enable/disable choices are preserved across updates
3. Administrators **enable or disable** flags through the Settings UI
4. The application checks flag state to **show or hide** features

### Managing Feature Flags

1. Navigate to **Settings > Feature Management > Feature Flags**
2. View all available feature flags with their current state
3. Toggle flags on or off as needed

Feature flags gate both backend API endpoints and frontend UI elements. When a flag is disabled:
- API endpoints return 404 or are unavailable
- UI pages and menu items are hidden
- Related functionality is completely inaccessible

## Organization Settings

### Team Member Roles

**Team Member Roles** define the roles that employees can hold when assigned to a team (e.g., Tech Lead, Engineer, Scrum Master, QA). Each organization defines its own set of roles.

Roles have:
- **Name** — A unique name (up to 128 characters)
- **Active** — Whether the role is available for assignment

Managing roles:
1. Navigate to **Settings > Organization > Team Member Roles**
2. Use **Create Team Member Role** to add new roles
3. Edit a role's name from the row menu
4. **Deactivate** a role to prevent it from being assigned to new members (existing assignments are unaffected)
5. **Delete** a role — only possible if no team members are currently assigned to it

## Work Management Settings

### Work Types

**Work Types** define the categories of work items available in the system (e.g., Epic, Feature, User Story, Task, Bug). See [Work Management](../work-management/index.mdx) for details on how work types are used.

### Work Statuses

**Work Statuses** represent the states within a workflow (e.g., New, Active, Resolved, Closed). Each status belongs to a **Work Status Category** that normalizes statuses across workflows:

| Category | Purpose |
|----------|---------|
| **Proposed** | Work has been proposed but not started |
| **Active** | Work is currently being performed |
| **Done** | Work has been completed |
| **Removed** | Work has been removed without completion |

### Work Processes

**Work Processes** define the configuration of work types, workflows, and hierarchies used within workspaces. See [Work Management](../work-management/index.mdx) for details.

## PPM Settings

### Expenditure Categories

**Expenditure Categories** classify project spending for financial tracking and reporting. Categories have:
- **Name** and **Description**
- **Capitalizable** flag — Whether the expenditure can be capitalized
- **Requires Depreciation** flag — Whether depreciation tracking is needed
- **Accounting Code** — Code for integration with accounting systems
- **State** — Proposed, Active, or Archived

### Project Lifecycles

**Project Lifecycles** define the template of phases that projects go through. See [Project Portfolio Management](../ppm/index.mdx) for details.

## Scoring Models

**Scoring Models** define formula-driven ways to score and rank work consistently — for example, prioritizing a backlog or portfolio projects by WSJF. Each model combines rated **criteria** with named **output formulas**, designating one as the primary score, and includes a **Test** tab for previewing results without saving. Active models can be assigned to [PPM portfolios](../ppm/portfolios-programs.mdx#portfolio-project-scoring). See [Scoring Models](./scoring-models.mdx) for the full guide. Configured in **Settings → Scoring → Scoring Models**.

## Integrations

Wayd integrates with external systems to synchronize data. Integrations are configured through **Connections**.

### Ownership Model

When data is synchronized from an external system, the objects created in Wayd have a **Managed** ownership. This distinguishes them from objects created directly in Wayd:

- **Owned** — The object is created and managed within Wayd. Users can edit it freely.
- **Managed** — The object is owned by an external system and is **read-only** in Wayd. Changes must be made in the source system and will be reflected in Wayd on the next sync.

Managed objects can still be associated or linked to owned items. For example, a [work item](../work-management/work-items.mdx#work-items) synced from Azure DevOps (managed in Wayd) can be linked to a team [objective](../planning/planning-intervals.mdx#objectives) owned by Wayd.

The ownership model applies to [workspaces](../work-management/work-items.mdx#workspaces), [work items](../work-management/work-items.mdx#work-items), [work processes](../work-management/work-configuration.mdx#work-processes), [workflows](../work-management/work-configuration.mdx#workflows), and [sprints](../planning/sprints.mdx).

### Connections

A **Connection** represents a configured integration with an external application via a Wayd **Connector**. Each connection includes authentication and endpoint configuration needed to establish the link.

A **Connector** is a configurable integration type that lets a specific external service plug into Wayd. Each connector belongs to a **Category** (Work Sync, People Sync, AI Provider) that determines how Wayd uses it. Multiple connections for the same connector type can be configured — for example, when an organization has multiple Azure DevOps instances, or multiple Entra tenants after an acquisition.

**Common actions on a connection:**

- **Edit** — Update name, description, or credentials. The masked secret can be left as-is to keep the existing value.
- **Activate / Deactivate** — Toggle whether the connection participates in sync runs. Inactive connections are excluded from every runner (work-sync, people-sync) and from "Sync Now" — the API rejects sync attempts on an inactive connection.
- **Sync Now** — Triggers a manual sync for that connection only. Available on Work Sync and People Sync connectors. Routes to the right runner based on the connector's category.
- **Sync History** — Tab on the connection detail page showing past runs with their status, duration, and per-run details (workspaces and work items for Work Sync; employees fetched and upserted for People Sync).
- **Delete** — Removes the connection. Past sync history rows are preserved by design.

### Azure DevOps Boards

The Azure DevOps Boards connector enables one-way synchronization of work data from Azure DevOps into Wayd. **Category:** Work Sync.

**Data mapping:**

| Azure DevOps | Wayd |
|-------------|------|
| Project | Workspace |
| Work Item | Work Item |
| Work Item Type | Work Type |
| Area | Team ↔ Work Item association |
| Iteration | Sprint / Planning Interval association |

**Configuration requirements:**
- **Organization** — The Azure DevOps organization name
- **Personal Access Token** — A PAT with Work Items Read scope

**Business rules:**
- Each Azure DevOps organization can only be linked by one connection
- Synchronized items have **Managed** ownership and are read-only in Wayd
- Sync is controlled by workspace-level integration state (active/inactive)
- Background jobs (Hangfire) perform synchronization on a recurring schedule

### Entra (Microsoft Entra ID)

The Entra connector synchronizes people (employees and contingent workers) from Microsoft Entra ID into Wayd's `Employee` records via Microsoft Graph. **Category:** People Sync.

**Capabilities:**
- Pull users from a single Entra tenant
- Optionally scope to a single Entra group (transitive members)
- Optionally include disabled accounts
- Upsert into Wayd `Employee` records, preserving the manager hierarchy
- Deactivate Wayd employees no longer present in the latest payload

**Configuration requirements:**
- **Tenant ID** — The Entra tenant's directory ID
- **Client ID** — Application (client) ID of the Entra app registration used to authenticate
- **Client Secret** — Application client secret (encrypted at rest, masked on display)
- **All Users Group Object ID** — *optional.* When set, only users transitively in this group are synced. Leave blank to sync all member users in the tenant.
- **Include Disabled Users** — *optional, default off.* When on, disabled accounts are included.

The Entra app registration needs the `User.Read.All` Microsoft Graph application permission (plus `GroupMember.Read.All` if you scope to a group). Both require admin consent.

**Business rules:**
- Multiple Entra connections are allowed — useful when integrating from more than one Entra tenant (for example, after an acquisition before tenant consolidation).
- People sync is always full — every run fetches the complete user set and reconciles against `Employee`.
- Employees not in the latest payload are automatically deactivated.
- Manager hierarchy is built from each user's Entra `manager` reference.

### AI Providers

AI provider connectors (Azure OpenAI, OpenAI) are outbound LLM clients. They don't sync data into Wayd — features that need AI access pick up the active connection at request time. **Category:** AI Provider.

## Background Jobs

Wayd uses **Hangfire** for background job processing. The background jobs dashboard is available at **Settings > Background Jobs** and provides:
- List of recurring jobs and their schedules
- Job execution history
- Manual job triggering

Background jobs are primarily used for:
- **Work Sync** — Synchronizes work data from all active Work Sync connectors (Azure DevOps today). Differential and Full variants are available.
- **People Sync** — Synchronizes people data from all active People Sync connectors (Entra today).
- **Portfolio Rank Rebalance** — Maintenance job that re-spaces dense project ranks within portfolios after repeated drag-to-rank operations.
- Other periodic maintenance tasks

## Estimation Scales

> This feature is behind the `planning-poker` feature flag.

**Estimation Scales** define the sets of values used for story point estimation in Planning Poker sessions. Scales are configured in **Settings > Planning > Estimation Scales**.
